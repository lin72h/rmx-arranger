# li-1010 — case-insensitive OpenZFS (macOS-like; phased, design-first)

li-1010 | tier: L1i | milestone 1 (1.0-preview) constituent | status: **HELD — design-first** | owner: Coordinator 2026-06-25

END STATE (long arc): whole-system case-insensitive filesystem like macOS, for app/userland compatibility
and the dogfood north star.

GOVERNING CONSTRAINT (Coordinator 2026-06-25): **FreeBSD base is authored case-sensitive by default —
turning the ROOT filesystem case-insensitive is expected to break boot if done naively.** Whole-system
insensitivity is therefore a *base-remediation* problem (audit + fix case-colliding paths in base, boot
path), NOT a property flip. This makes the whole-system end state a careful long-arc effort, almost
certainly **post-preview** — confirm gate scope.

HARD ZFS FACT: `casesensitivity` (+ `normalization`, `utf8only`) are **create-time only — immutable on an
existing dataset.** So every phase = **create a new insensitive dataset + migrate data** (send/recv or
copy), never an in-place toggle. Root conversion = reinstall onto a new boot dataset.

PIN BEFORE CREATING THE FIRST DATASET (cannot change later):
- `casesensitivity`: **`mixed`** (case-preserving + insensitive lookups, closest to APFS) vs `insensitive`
  (blunt). Lean `mixed`.
- `normalization`: `formC` vs `formD` (macOS HFS+ used an NFD variant) — only bites with non-ASCII names.

PHASES (each carefully designed before execution):
- **P1 (near-term candidate, low blast):** `/Users/<user>` on a dedicated insensitive dataset. Highest
  compat payoff, smallest risk. Plausibly in-preview scope.
- **P2:** `/usr/local/*` — only AFTER a case-collision audit of ports/pkg payloads.
- **P3 (long-arc / post-preview):** whole-system / root — requires base case-collision remediation + new
  boot dataset; the boot-break risk above lives here.
