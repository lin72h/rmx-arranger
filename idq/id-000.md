# id-000 — IDQ (backlog) overview + index

Work hierarchy: **`L1i → IDQ → ROB`** (most abstract → in-flight; aliases roadmap/backlog/op;
see [terminology.md](../terminology.md) §6). **L1i** (the [roadmap](../roadmap.md), `li-NNN`)
names *what usable 1.0 means and the milestone ladder `li-001`…`li-006`*; each L1i instruction is
*decoded* into **IDQ** items here (`id-NNN`); an IDQ item is *fetched* into **ROB** op(s) (`op-NNN`).

The **IDQ** is the decoded instruction stream *upstream of fetch* — items identified but **not
yet fetched into the ROB**. In the CPU/OoO model the rest of the workflow uses:

- **IDQ (`id-NNN`)** = decoded uops queued, not yet fetched. Planning-level. No pipeline, no
  agent assigned, no evidence spend.
- **ROB (`op-NNN`)** = an instruction that has been *fetched → issued*; now in flight in the
  reorder buffer (executing / waiting / retiring), owned by a pipeline + agent.

**Promotion = fetch.** When an `id-NNN`'s preconditions are met and the Coordinator decides
to work it, it is *fetched*: decomposed into one or more `op-NNN` and issued. The IDQ item
then points at the op(s) that carry it and is no longer planning-level.

`id-NNN` is zero-padded, sequential, never reused (same discipline as `op-NNN`). `id-000`
is this index; entries start at `id-001`.

## State vocabulary (mirrors the op states)

| state | meaning (IDQ) |
|---|---|
| **WAITING** | preconditions unmet — names what it waits on: a Coordinator decision, or an in-flight `op-NNN` that must retire first. Analogous to an op in the reservation station. |
| **READY** | preconditions met; fetchable now — ready to be decomposed into op(s) and issued. |
| **FETCHED → op-NNN** | promoted out of the backlog into flight; tracked thereafter by its op id(s) (the ROB owns it). Kept here with the pointer. |
| **RETIRED** | its op-chain completed. Terminal; kept for history. |
| **DROPPED** | abandoned before promotion (decision reversed / superseded). Kept for history with the reason. |
